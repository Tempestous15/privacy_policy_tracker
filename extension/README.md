# ClipPri (extension)

Fully local. No account, no server -- everything below runs in your browser.

## Load it for testing

**Chrome / Edge**
1. Go to `chrome://extensions` (or `edge://extensions`), enable Developer mode.
2. "Load unpacked" → select this `extension/` directory.

**Firefox**
1. Go to `about:debugging#/runtime/this-firefox`.
2. "Load Temporary Add-on…" → select `manifest.json` in this directory.

The classifier and ToS;DR lookup work fine in Firefox. The on-device AI
summary (WebLLM, via WebGPU) is effectively Chrome/Chromium-only right now
-- Firefox doesn't support WebGPU in extension background contexts yet, so
"Get AI summary" will fail there with an inline error rather than working.

## What runs where

- **`redflags-engine.js`** (generated by `../classifier/`) -- lexicon-based
  red-flag classifier, runs instantly on policy text, no network call.
- **`discovery.js`** -- finds a site's privacy-policy URL by scanning the
  current page's links (via `chrome.scripting.executeScript`) and falling
  back to common paths (`/privacy`, etc). The policy is very often on a
  *different domain* than the site itself (e.g. google.com's policy lives at
  policies.google.com), so fetching it needs the broad `host_permissions`
  below, not just `activeTab`.
- **`tosdr.js`** -- queries the public [ToS;DR](https://tosdr.org/) API
  directly for community-reviewed ratings on well-known sites. The only
  network request this extension makes to a third party.
- **`storage.js`** -- saved sites, kept in `chrome.storage.local` only.
- **`webllm-client.js`** / **`background.js`** -- an on-device LLM
  ([WebLLM](https://github.com/mlc-ai/web-llm), via WebGPU) that generates
  the plain-English summary. The model runs in the MV3 service worker
  (`background.js`); `webllm-client.js` is the popup-side proxy that talks
  to it over a `chrome.runtime.Port`. Both files are generated -- see below.

## Rebuilding the WebLLM bundles

`background.js` and `webllm-client.js` are committed, generated artifacts
(same pattern as `../classifier/dist/redflags-engine.js`) -- bundled from
`webllm/background-src.js` / `webllm/client-src.js` via esbuild, since
`@mlc-ai/web-llm` is published as an ES module meant for a bundler, not a
plain `<script src>`. Regenerate after editing either source file or bumping
the `@mlc-ai/web-llm` version in `webllm/package.json`:

```bash
cd webllm && ./build.sh
```

To change the on-device model, edit `MODEL_ID` in `webllm/client-src.js`
(any id from WebLLM's `prebuiltAppConfig`) and rebuild.

## Required manifest bits

- `content_security_policy.extension_pages` must include `'wasm-unsafe-eval'`
  -- without it, WebLLM (and, if added, Pyodide) fail at WebAssembly
  instantiation. This is *not* on by default; it must be declared explicitly.
- `host_permissions` is broad (`http://*/*`, `https://*/*`) -- needed
  because a discovered policy URL can be on any domain (see `discovery.js`
  above), plus it covers `api.tosdr.org` (ratings) and `huggingface.co` /
  `raw.githubusercontent.com` (where WebLLM fetches model weights/WASM
  runtime from on first use).
- `background` declares **both** `service_worker` and `scripts` pointing at
  the same `background.js`: Chrome (MV3) uses `service_worker` and ignores
  `scripts`; Firefox's MV3 support doesn't implement service workers yet and
  uses `scripts` instead (an event page), ignoring `service_worker`. Without
  both keys present, Firefox refuses to load the extension at all
  ("background.service_worker is currently disabled").

## Known limitation

The on-device summary needs a real GPU with WebGPU compute-shader support.
Under a software/virtualized GPU (e.g. SwiftShader, as seen in some headless
CI environments) the model downloads and loads correctly but shader
compilation fails -- that reflects the environment's actual GPU, not a bug,
and isn't something the extension can detect or work around.
